## ⚠️ 已知问题

### 高优先级问题

#### 1. LLM输出清理不完全
**现象**: 
- `<think>` 标签仍然泄露到报告中
- 重复的提示词（如评论#3重复7次）
- 思考过程文本未完全移除

**示例**（KAN-7报告）:
```
<think>
1. **Analyze the Request:**
   * Input: Two Jira Issue descriptions...
   * Task: Analyze the two issues...
</think>
```

**根本原因**: 
- qwen3.5-0.8b模型倾向于输出思考过程
- 正则表达式清理不够强力
- 嵌套标签处理不完善

**已采取措施**:
- ✅ 增强 `clean_llm_output` 函数（commit 644ec42）
- ✅ 添加 while 循环处理嵌套标签
- ✅ 强化提示词移除模式

**待验证**: 需要重新测试验证效果

#### 2. 行动建议完全为空
**现象**: 所有测试用例的"行动建议"章节都是空白

**可能原因**:
- ActionRecommender的提示词不够明确
- LLM响应解析失败
- 响应被过度清理

**待排查**: 需要添加调试日志查看原始LLM响应

#### 3. 闭环检查结果矛盾
**现象**（KAN-7）:
```
**闭环状态**: ❌ 未闭环
- **根因识别**: ✗ 
- **修复方案**: ✗ 
- **验证测试**: ✗ 
**结论**: **已闭环**  ← 矛盾！
```

**根本原因**: ClosedLoopChecker的解析逻辑有bug

#### 4. 评论分析质量不稳定
**现象**:
- 有时返回完整分析（评论#2）
- 有时只返回提示词（评论#1）
- 有时重复提示词（评论#3）

**示例**（KAN-7评论#1）:
```
**分析**:
请只输出三个维度的结论性分析结果（JSON），不要有任何其他内容。
```

## 🎓 经验教训

1. **小模型的局限性**: qwen3.5-0.8b 在复杂任务上表现不稳定
2. **提示词工程的重要性**: 需要反复迭代优化
3. **输出清理的必要性**: 小模型容易泄露思考过程
4. **测试驱动开发**: 应该先写测试再实现功能
5. **渐进式开发**: Mock LLM → 真实LLM 的策略是正确的

---

**总结**: 核心功能已实现，但需要解决LLM输出质量问题才能投入生产使用。建议优先修复P0问题，然后考虑升级到更强的模型。
