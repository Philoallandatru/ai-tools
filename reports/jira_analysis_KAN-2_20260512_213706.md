# Jira 深度分析报告: [KAN-2] [SV][SSD1250][Sanitize] Block Erase 期间下发 NVM Reset 导致下次上电设备进入只读模式

## 📋 问题摘要

**客户名称**: Micron
**测试项目**: SSD1250, Sanitize
**测试平台**: Platform: AMD Ryzen 7000 Series reference platform; OS: Ubuntu 22.04 LTS; Form Factor: M.2 2280

**测试步骤**:
1. 发起 Sanitize (Block Erase) 操作 nvme sanitize /dev/nvme0 -a 2
2. 轮询 Sanitize Status (Log Page 81h)
3. 在进度达到 30%-50% 期间，通过 PCIe 写入 Controller Configuration (CC.EN=0 -> 1) 触发 NVM Reset
4. 等待设备重新枚举并 Ready
5. 尝试进行 4K Random Write

**根因**: Sanitize (Block Erase) 操作期间处理 NVM Subsystem Reset 时，状态机未正确清理残留的背景擦除队列，导致下次上电时 FTL 映射表重建（Rebuild）校验失败，触发自我保护进入只读模式
**修复方案**: 在 Sanitize Block Erase 阶段检测到 NVM Reset 时，先完成当前 block 的 erase，再进入 reset handler

## 根因分析

**直接原因**: 在 Sanitize 中断期间，NVM Reset 触发的状态机复位逻辑未能彻底清理残留的后台擦除队列和元数据锁，导致 FTL 重建时校验失败并误判为只读模式。

**深层原因**: NVMe Subsystem Reset 协议未强制要求清空当前进行中的媒体验证（Media Verification）及后验证释放（Post-Verification Deallocation）状态，而 FTL 在重新上电时缺乏针对这些“挂起”状态的恢复机制。

**触发条件**: 必须在 Sanitize 进度达到特定阈值（30%-50%）且设备处于后台擦除活跃期时，通过 PCIe 下发 NVM Reset 命令来打断当前操作。
